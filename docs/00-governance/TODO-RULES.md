# TODO Rules

## 1. Purpose

本文档定义根目录 `TODO.md` 的任务格式、协作用法和删除规则。

目标是让 `TODO.md` 只承载未关闭任务队列，不承载方案正文、完成清单、长篇论证或未经审阅的执行许可。

## 2. Scope

当前范围：

- 根目录 `TODO.md`
- 宏观任务与执行任务的边界
- 任务拆解、人机审阅和执行许可
- `TODO.md` 的固定格式
- 任务完成后的删除、拆分和收窄
- 任务收口时的测试、文档、commit 和 PR 检查

不在范围内：

- 不替代具体模块需求文档
- 不替代架构红线或实现规则
- 不替代测试写法教程
- 不替代 Git 基础使用教程
- 不记录已完成任务清单

## 3. Bounded Context

`TODO.md` 是未关闭任务面板。

本仓库把“任务完成”定义为一个闭环，而不是单纯代码已改完。代码、测试、文档、`TODO.md` 清理和提交必须收敛成可解释的变更链路。

`TODO.md` 固定只回答：

1. 当前还有什么未关闭任务。
2. 每个任务的处理动作是什么。
3. 每个任务完成时如何判断。

完整方案、规则背景和执行细节必须放在对应治理文档、`HOW-TO` 或专项设计文档中。`TODO.md` 只引用这些文档，不复制这些文档。

TODO、commit 和 PR 固定组成一套协作模型：

- TODO 是未关闭任务队列，表达当前还要做什么。
- Commit 是工程判断记录，表达一次具体判断或小步能力变化。
- PR 是阶段性交付边界，必须完整、可编译、可测试，并完成文档、TODO 和 RUNBOOK 收口。
- Commit 可以是中间态，但必须有明确判断含义，不得是随机保存点。
- Commit 不强制每一笔都通过全量验证；涉及可运行入口的 commit 必须运行该判断的最小相关验证。
- PR 必须通过当前阶段对应的完整验证入口。

## 4. TODO Types

`TODO.md` 承载三类任务：

- 拆解任务：目标已经确定，必须按指定 Runbook 或治理文档拆成执行任务。
- 执行任务：已经达成共识，可以进入实现、测试、文档同步和提交。
- 待讨论项：需要人类先做决策，不能直接执行。

拆解任务和待讨论项不是执行许可。拆解并经人类审阅后的执行任务，才是执行入口。

## 5. File Structure

`TODO.md` 固定使用以下结构：

```markdown
# TODO List

## 说明

- `TODO.md` 是未关闭任务面板，不是完成历史。
- 宏观任务必须先讨论边界，再拆解为可执行 TODO。
- 已完成任务必须删除，不在 `TODO.md` 中打勾长期保留。
- 完成历史保留在 commit 或 PR 中。

## 当前任务项

- [ ] `范围对象`：任务标题
    - 任务类型：执行任务
    - 依据文档：`docs/.../XXX.md`
    - 范围对象：精确文件路径、模块或对象
    - 处理动作：一句话说明要做什么
    - 验收点：可检查的完成结果
    - 重要度：N/10

## 待审阅任务项

- [ ] `范围对象`：任务标题
    - 任务类型：拆解任务或执行任务
    - 依据文档：`docs/.../XXX.md`
    - 范围对象：精确文件路径、模块或对象
    - 处理动作：一句话说明要做什么
    - 验收点：可检查的完成结果
    - 重要度：N/10

## 待讨论项

- [ ] 需要决策的问题
    - 任务类型：待讨论项
    - 关联任务：任务名或范围对象
    - 决策要求：需要人工确认的具体问题
    - 重要度：N/10
```

没有任务时，对应章节可以为空，但不得记录已完成任务清单。

## 6. Task Item Rules

任务固定使用 `- [ ]`。

执行任务必须包含：

- `任务类型`
- `依据文档`
- `范围对象`
- `处理动作`
- `验收点`
- `重要度`

规则：

- `范围对象` 必须能稳定定位模块、文件或治理域。
- `依据文档` 必须指向当前任务的稳定规则、Runbook 或需求来源。
- 一个执行任务只表达一个主动作。
- `处理动作` 只写一个明确动作。
- `验收点` 只写可检查结果。
- `重要度` 使用 `N/10`。
- 单项任务文件过多时必须继续拆分。

不得在任务项中写：

- 完整方案
- 长篇背景
- 预设提交信息
- 长命令教程
- 已完成任务清单
- 与该项无关的后续步骤

## 7. Collaboration Rules

写入 TODO 前，固定判断：

1. 该任务是否已经足够重要，值得进入 `TODO.md`。
2. 该任务是否能用一句话说明问题或目标。
3. 该任务是否能明确归类为拆解任务、执行任务或待讨论项。
4. 该任务是否不是低置信度观察项。
5. 该任务是否存在明确的下一步动作、拆解依据或决策问题。

拆解执行任务前，固定判断：

1. 拆解任务或待讨论项是否已经经过人机讨论。
2. 拆解后的 TODO 是否需要人工审阅。
3. 每个执行 TODO 是否能独立验收。
4. 每个执行 TODO 是否能对应到代码、测试、文档或提交动作。
5. 执行顺序是否合理。

未经人工确认的拆解任务和执行任务固定放入 `待审阅任务项`。AI 不得执行 `待审阅任务项`。

## 8. Deletion Rules

完成任务后固定处理：

- 完成整项：删除该 TODO。
- 完成部分：收窄为剩余未完成内容。
- 发现拆分错误：重写为正确的待审阅任务。
- 待讨论项完成决策：删除待讨论项；如仍需执行，新增明确执行任务或拆解任务。

已完成任务不得在 `TODO.md` 中打勾长期保留。完成记录保留在 commit 或 PR 中。

删除或收窄 TODO 必须发生在真正完成该 TODO 的 commit 中。

如果 commit 只是中间判断，不删除 TODO。

阶段任务完成前，相关 TODO 必须清空或收窄为剩余未完成内容。

## 9. Closure Rules

任务完成后，进入收口前固定先判断：

1. 当前任务的实现目标是否已经达到可运行、可检查状态。
2. 当前改动是否仍存在未完成的剩余范围。
3. 当前改动是否影响测试、文档或任务入口。
4. 当前工作区中是否混入了其他不相关修改。
5. 当前 commit 是否能够用一句明确的话解释“这次到底记录了什么判断”。

只要以上任一问题不能明确回答，就不得进入收口。

收口时固定先判断 `TODO.md` 对应任务项是否真的完成：

- 任务目标已经完成，且没有剩余未实现范围：删除对应 `TODO.md` 项。
- 任务只完成一部分，但剩余范围仍清楚存在：不要删除；将任务拆分或收窄为剩余未完成内容。
- 当前只是局部验证通过，但需求或边界尚未稳定：不要删除；保持原任务项不动。

## 10. Test Check Rules

收口前必须检查当前改动影响了哪些测试层次。

最小检查清单：

- 是否新增或改变公开入口、协议签名、校验或异常响应。
- 是否改变运行时编排、状态流转、幂等、跨模块调用或结果模型。
- 是否改变规则解析、规则匹配、规则执行或规则优先级。
- 是否改变 IDE、MCP、hook 等外部集成行为。
- 是否改动了架构规则、命名规则或其他可能影响静态门禁的内容。

判断结果：

- 行为变化已发生：同步补或更新对应测试。
- 只有纯机械改名、注释、格式整理，且不影响行为和规则：可以不新增测试，但必须确认没有误触行为边界。

## 11. Document Sync Rules

收口前必须检查当前改动是否要求同步文档。

最小检查清单：

- 模块需求变化：检查 `10-requirements/`。
- 治理规则或固定做法变化：检查 `docs/00-governance/*.md`。
- 接口、协议或契约变化：检查 `20-interfaces/`。
- AI 默认读取入口或任务路由变化：检查 `docs/AGENTS.md`。

判断原则：

- 改动影响文档定义的固定口径时，必须同步文档。
- 仅代码实现补齐、但不改变既有文档口径时，不为“看起来完整”而顺手改无关文档。

## 12. Commit Rules

commit 粒度固定围绕工程判断组织。一个 commit 表达一次具体判断、小步能力变化、测试锁定、配置装配或文档规则落点。

commit 可以是阶段任务的中间态，但必须满足：

- 判断含义明确。
- 改动范围可解释。
- 不混入无关修改。
- 不故意留下不清楚的破坏。
- 涉及可运行入口时，运行最小相关验证。

commit 不要求代表完整阶段任务，也不要求每一笔都通过全量验证。PR 才是完整可编译、可测试和文档收口的交付边界。

可以放在同一个 commit 的内容：

- 同一判断的代码、测试、文档同步。
- 真正完成对应 TODO 时，对应 `TODO.md` 项删除、拆分或收窄。
- 为同一判断服务的命名或装配修正。

必须拆成多个 commit 的内容：

- 不同模块的独立能力改动。
- 同一任务中彼此无直接因果关系的两类判断。
- 纯治理补丁与功能改动，如果它们不是同一个闭环。

`commit message` 必须同时表达改动类型、模块或治理域、具体能力变化。

本仓库固定格式：

`Type(scope): 中文说明`

检查清单：

- `Type` 是否准确表达这是 `Feat`、`Fix`、`Docs`、`Test`、`Refactor` 等哪类变化。
- `scope` 是否能稳定定位模块或治理域。
- 中文说明是否点出具体能力，不写空泛的“调整”“修改”“优化”。

## 13. PR Rules

PR 固定表示阶段性交付边界。

PR 必须满足：

- 阶段任务完整。
- 可编译。
- 可测试。
- 对应验证命令已运行。
- 行为、接口或流程变化已同步文档。
- 相关 TODO 已清空或收窄为剩余未完成内容。
- 已完成且无剩余价值的 RUNBOOK 已清理。
- PR 描述记录验证命令和结果。

PR 的验证入口固定由 [`../40-readiness/PR-WORKFLOW.md`](../40-readiness/PR-WORKFLOW.md) 和 `scripts/verify-all.sh` 承载。

## 14. Verify Protocol

自动化 testcase 固定使用四段式验证协议：

1. `Prepare`：准备测试数据、临时目录、fixture、环境变量、端口、server stub 或依赖安装。
2. `Execute`：执行被测命令、脚本、HTTP 调用或构建命令。
3. `Assert`：验证退出码、stdout、stderr、HTTP response、生成文件或数据结构。
4. `Restore`：删除临时目录、恢复被改动文件、停止后台进程、清理环境变量。

验证脚本必须保证失败路径也执行 `Restore`。

验证脚本固定分两层：

- `scripts/verify-all.sh`：总入口，只负责编排各模块验证。
- `scripts/verify-*.sh`：模块或 testcase 入口，负责自己的 `Prepare`、`Execute`、`Assert` 和 `Restore`。

所有 `scripts/verify-*.sh` 固定满足：

- 使用 `set -euo pipefail`。
- 从脚本位置定位 repo root。
- 输出当前正在验证的模块或 testcase。
- 失败时返回非 0。
- 不依赖调用者当前目录。
- 不依赖用户全局配置。
- 不把临时数据写入未声明的位置。
- 执行完成后恢复由脚本创建或修改的数据。
